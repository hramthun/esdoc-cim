A real world earth system property (e.g. sea surface temperature).  This is the property itself, independent of any process that uses it or any connection in which it is involved.<br /></p><p><br /></p><p>The property is abstracted from implementation details (which are represented in Connection and PropertyRole), and from the data representation (which is represented in DataContent).<br /></p><p><br /></p><p>This type inherits from FC_FeatureAttribute, so attributes for name, description, units and values are inherited as well.<br /></p><p><br /></p><p>*Is it valid to connect the process to the real world earth system property with an association class to contain details of the association between the process and the property, and to have a connection to the DataContent in a separate association?  The possible problem with this is that the property might have several different data representations (in DataContent), and the process and the connection might be interested in a specific data content, rather than the generic property.  In this case, the Connection and PropertyRole association classes should be connected to DataContent instead of RealWorldEarthSystemProperty.<br /></p><p><br /></p><p>*The cardinality of the observedProperty has been changed from O&M, which provides cardinality 1 (only one observed property per observation).  This is consistent with the ISO 19110 (FTC) constructs that also represent this relationship.  That is, the status of M4A_Observation as an instance of FC_FeatureOperation and MF4_RealWorldEarthSystemProperty as an instance of FC_FeatureAttribute allows the observedValuesOf association role to be used to connect observations (simulations) to the properties they observe.  <br /></p>
